Coarse-Grained API Design — Justification

✓ Our Design (Coarse-Grained)

  • 1 HTTP call carries everything the external service needs
  • Recipient info, appointment detail, and notification type bundled in one payload
  • External service is self-sufficient — never queries back into our database
  • Failure of the notification service does not roll back the booking transaction
  • Clear ownership: Appointment Scheduler owns booking; Notification Service owns delivery

✗ Fine-Grained Alternative (Rejected)

  • POST /notif/jobs → create job, get jobId
  • PUT /notif/jobs/{id}/recipient → set who receives it
  • PUT /notif/jobs/{id}/template → choose template
  • POST /notif/jobs/{id}/send → actually send
  • GET /notif/jobs/{id}/status → poll for result
  • 5× more round-trips; tight coupling to internal state machine of the external service

Design Principles Applied

  • Low coupling across the boundary: The Appointment Scheduler does not know (or care) whether the Notification Service uses SendGrid, Twilio, or a log file. It calls one endpoint and receives a status.
  • High cohesion within each service: The Notification Service owns all delivery logic internally. The Scheduler owns all booking logic internally. Neither reaches into the other's data store.
  • Fail-safe boundary: The NotificationClient catches all exceptions and logs them as WARN. A notification failure never propagates back to cancel a completed booking.
  • Idempotent-friendly: The caller can retry the single POST if it times out, without risk of partial state on the external side.
  • Evolvability: Adding SMS, push, or WhatsApp delivery in the future requires only a change to channel in the payload — no new endpoints needed on the Scheduler side.

Responsibility Boundary

ConcernOwnerRationale
Slot locking & double-booking prevention AppointmentService (SERIALIZABLE transaction) Must stay atomic with the DB write; cannot be delegated
Booking persistence AppointmentRepository Internal data concern; external service has no write access
Assembling the notification payload NotificationService + NotificationClient Translates internal Appointment model to the external contract
Choosing delivery channel & template MockNotificationController (external) Delivery detail is the external service's concern — hidden from Scheduler
Generating a notification tracking ID MockNotificationController (external) Owned by the service that performs the delivery
Retry & failure isolation NotificationClient (catch block) Network errors must not propagate into the booking transaction
